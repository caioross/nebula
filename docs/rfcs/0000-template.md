## Summary

Three or four sentences. A reader should know what you are proposing and why before deciding to read on.

## The problem

What is wrong or missing today, concretely. If the problem only exists for a hypothetical user, say so — hypothetical problems get hypothetical priority.

## The proposal

Precise enough that two people would implement the same thing. Describe observable behavior first, mechanism second. If the proposal touches the canvas, write the actual sentences the user and Nebula would exchange.

## Fit with the founding constraints

Address each constraint the proposal comes near (see GOVERNANCE.md). "Does this add anything a first-time user must understand before writing?" gets an explicit answer here.

## Costs

What gets slower, heavier, more complex or riskier. Every real proposal has this section with real content. An RFC that claims no costs is returned unread.

## Alternatives considered

Including "do nothing", which is often strong.

## Open questions

What you don't know yet, honestly.
